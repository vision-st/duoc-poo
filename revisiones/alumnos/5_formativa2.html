Retroalimentación general

Querido equipo: su código demuestra una buena comprensión de los principios de herencia y polimorfismo. La estructura general está clara y bien separada por responsabilidades. Se nota una intención de aplicar buenas prácticas y mantener legibilidad. Sin embargo, se observan algunas oportunidades de mejora, como una implementación algo rígida de la interfaz `IMostrable`, la falta de un uso más explícito de constructores sobrecargados en la clase abstracta `CuentaBancaria`, y un manejo que puede beneficiarse de más consistencia en la validación de datos.

Criterios CL L ML LI NL Observaciones
1. Clase abstracta con al menos dos atributos X `CuentaBancaria` cumple este criterio correctamente. Se definen `numeroCuenta` y `saldo` como atributos clave del modelo.
2. Tres subclases que heredan de la clase base X Se crean `CuentaAhorro`, `CuentaCorriente` y `CuentaCredito`, cada una con implementación coherente. Bien logrado.
3. Aplica una interfaz que define un método X La interfaz `IMostrable` es aplicada, pero con escasa profundidad. Podrían agregarse más usos que refuercen su utilidad y separación de lógica.
4. Sobrescribe método abstracto de clase base X Las subclases sobrescriben correctamente métodos abstractos como `consultarSaldo()` y `mostrarDatosCuenta()`, incluso si la lógica es básica.
5. Sobrecarga de constructores en clase abstracta X El constructor sobrecargado no se implementa completamente. Una sugerencia es agregar:
public CuentaBancaria(String numeroCuenta) { this(numeroCuenta, 0); }

Feed Up – ¿Dónde debe llegar el estudiante?

El objetivo de esta actividad es consolidar el uso de herencia, polimorfismo, interfaces y sobrecarga, simulando una estructura bancaria que permita gestionar clientes y cuentas a través de una jerarquía lógica de clases.

Feed Back – ¿Cómo se está desempeñando?

El grupo cumple con la mayoría de los requerimientos fundamentales. Las clases están correctamente organizadas, los métodos están bien aplicados y existe una clara intención de reflejar polimorfismo en las operaciones. El uso de la interfaz puede expandirse, y la sobrecarga de constructores quedó en un nivel básico. Muy buen trabajo, pero aún hay espacio para perfeccionar estructura y reusabilidad.

Feed Forward – ¿Cómo debe seguir?

Como siguiente paso, sería recomendable aplicar esta misma estructura en otro dominio (por ejemplo, una tienda de productos o un sistema de biblioteca), reforzando así la comprensión de la herencia. También sería beneficioso evitar repetir código y aplicar técnicas como delegación para mantener un diseño más limpio. Revisa además algunas guías de estilo Java para mantener consistencia entre clases.
