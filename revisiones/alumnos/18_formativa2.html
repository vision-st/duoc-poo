Actividad: Utilizando Herencia y Polimorfismo

Grupo 18 - JKARLA SANTIBANEZ GUTIERREZ y KEVIN CRISTOFER CARRASCO VICENCIO

El trabajo del grupo demuestra un dominio sólido de los conceptos esenciales de la Programación Orientada a Objetos. La estructura del código es clara y bien dividida: Cuenta funciona como clase abstracta, mientras que CuentaCorriente, CuentaAhorro y CuentaCredito heredan correctamente y personalizan su comportamiento. El uso de la interfaz InformacionCliente para estandarizar la visualización de datos muestra una buena comprensión del polimorfismo.

El código de interacción en BankBostonApp y Main es funcional, aunque algo extenso en algunas secciones. Sería ideal centralizar la lógica repetida (como validaciones o impresión de menús) para mejorar legibilidad. También se destaca el uso de constructores sobrecargados que permiten crear cuentas con o sin saldo inicial.

En general, la implementación es robusta, modular y con buen uso de los principios solicitados.
CriteriosCLLMLLINLObservaciones
1. Clase abstracta con al menos dos atributos X Cuenta tiene los atributos numeroCuenta y saldo, bien encapsulados y usados correctamente.
2. Tres subclases que heredan de la clase base X Cada clase (CuentaCorriente, CuentaAhorro, CuentaCredito) representa correctamente su tipo.
3. Aplica una interfaz que define un método X InformacionCliente permite aplicar un mismo método en diferentes clases, cumpliendo el objetivo del polimorfismo.
4. Realiza la sobreescritura de un método abstracto X Cada subclase implementa sus versiones específicas del método heredado, utilizando System.out.println para entregar la información.
5. Aplica sobrecarga de métodos en la clase abstracta X Se observa uso de sobrecarga de constructores en Cuenta para manejar distintos escenarios de inicialización.

Feed Up

El objetivo de esta actividad es aplicar herencia, abstracción, polimorfismo, sobrecarga, sobreescritura e interfaces en un sistema concreto. El foco está en estructurar el código de manera reutilizable, legible y extensible.

Feed Back

El grupo presenta una solución bien lograda, con separación de responsabilidades, claridad en el flujo y aplicación efectiva de los principios de la POO. El sistema tiene potencial para crecer, aunque podría beneficiarse de una mejor división del código de lógica y presentación.

Feed Forward

Continúen experimentando con la modularización: intenten separar las operaciones bancarias en clases de servicio o controlador. Esto no solo mejora la legibilidad, sino también la capacidad de hacer pruebas y mantenimiento a futuro. El libro "El Programador Pragmático" ofrece buenos consejos sobre refactorización progresiva.
